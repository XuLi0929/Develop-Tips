Preventing Data Loss: A Simple Automated Backup for Your Working Directory

Published: February 10, 2026 · 3 min read · Updated: February 12, 2026

If you're working on a remote server and worry about losing files due to accidental deletion or a crash, a quick local backup script can save you a lot of headaches. This guide walks you through setting up an automated backup of your current working directory in under five minutes.

This approach is perfect for development servers, staging environments, or any situation where you want a simple safety net without configuring a full backup solution like restic or borgbackup.

Step-by-Step Setup

Step 1 — Create the Backup Script

First, create a bash script that zips the entire current directory into a backup archive. This gives you a snapshot you can restore from if anything goes wrong.

#!/bin/bash
zip -r backup_v1.zip .

Save this as backup.sh in your working directory and make it executable with chmod +x backup.sh.

Step 2 — Add Automatic Scheduling

A single backup is helpful, but what you really want is continuous protection. Modify the script so it runs every 60 seconds and appends a timestamp to each backup filename. This way you always have a rolling history of your work.

#!/bin/bash
while true; do
    zip -r "backup_$(date +%Y%m%d_%H%M%S).zip" .
    sleep 60
done

This creates a new timestamped zip file every minute, so you never lose more than 60 seconds of work.

Step 3 — Run It Persistently

Finally, launch the script in the background using nohup so that the backup process continues running even if you disconnect from the server or close your terminal session.

nohup bash backup.sh > /dev/null 2>&1 &

That's it! Your automated backup is now running. You can verify it's working by checking for new zip files appearing in your directory after a minute or two.

Wrapping Up

This is one of the simplest ways to protect yourself from accidental data loss on a server. It takes less than a minute to set up and runs quietly in the background. For production environments you'll want something more robust, but for day-to-day development work, this gets the job done.
